# AI 主动触发记忆的回合末强制执行设计

- 日期: 2026-04-09
- 状态: 评审通过（会话确认）
- 范围: ai_end 中 form_memory 工具触发语义与自动触发门槛协同
- 非目标: 不改数据库结构、不新增迁移、不改 MemoryManager 的解析/重试协议

## 1. 背景与目标

当前记忆系统存在两类触发路径:

1. AI 在对话中主动调用 form_memory 工具。
2. 对话结束后根据消息数量达到阈值（当前 5 条）自动触发。

本次目标:

1. 当 AI 主动调用 form_memory 时，不在工具调用当下直接生成记忆。
2. 仅在当前回合结束后统一执行一次硬编码记忆生成。
3. 该强制执行不受最低轮数门槛限制。
4. 普通自动触发仍保持原 5 条门槛不变。

## 2. 方案对比与推荐

### 2.1 方案 A（推荐，技术债最低）

- form_memory 工具调用仅置位“本回合强制记忆”标记。
- 回合结束统一执行一次记忆生成。
- 强制路径绕过 5 条门槛。

优点:

1. 触发语义清晰，避免重复落库。
2. 改动面小，复用现有 MemoryManager 单一生成入口。
3. 便于未来扩展更多触发源。

### 2.2 方案 B

- 工具调用立即执行一次，回合结束再做去重判断。

缺点:

1. 容易出现双执行/双写风险。
2. 去重状态复杂，技术债中等。

### 2.3 方案 C

- 取消所有路径的最低门槛。

缺点:

1. 与需求边界冲突（仅 AI 主动触发放开）。
2. 易造成过度写入，技术债高。

结论: 采用方案 A。

## 3. 架构与职责边界

### 3.1 触发层与执行层分离

- 触发层: handlers + client，负责“意图与调度”。
- 执行层: MemoryManager，负责“生成、校验、重试、保存”。

### 3.2 form_memory 工具语义调整

- handlers 接收 form_memory tool_call 后，不直接调用 MemoryManager。
- 返回“已登记，本回合结束后执行”的工具结果。
- 将执行意图以本回合状态标记方式传递给 client。

### 3.3 回合末统一执行点

在 client 的“无 tool_calls，准备结束回合”分支:

1. 若命中强制标记，立即调用现有 form_memory 硬编码执行链并绕过门槛。
2. 否则按原逻辑检查 `(synced_history_count + 2) >= 5` 再自动触发。

## 4. 数据流与状态机

### 4.1 新增瞬时状态位

- 建议字段: `_force_memory_after_turn: bool`
- 默认 false，仅当前回合有效，不持久化。

### 4.2 状态流转

1. 收到 AI form_memory tool_call -> 置位 true。
2. 回合末裁决:
   - true: 执行一次记忆生成（忽略门槛）。
   - false: 走原 5 条门槛逻辑。
3. 回合末无论成功失败都清零为 false。

### 4.3 幂等与去重

- 同回合多次 form_memory tool_call 仅置位一次。
- 回合末最多执行一次记忆生成。

## 5. 异常处理与边界约束

1. 强制触发只绕过轮数门槛，不绕过 user_id 和消息非空等基础前置条件。
2. 记忆生成失败沿用现有重试与失败返回，不中断主聊天流程。
3. 回合中断场景必须清理标记，防止污染下一回合。
4. 强制路径与普通自动路径互斥，避免双写。

## 6. 测试策略（TDD）

### 6.1 RED: 先补失败用例

1. handlers 单测
   - AI 调用 form_memory 时不直接执行 MemoryManager。
   - 返回文案包含“回合末执行”语义。

2. client 单测
   - 强制标记为 true 时，未达 5 条也会在回合末执行 form_memory。
   - 同回合多次调用仍只执行一次。
   - 回合结束后标记被清零。

3. integration 回归
   - 未设置强制标记时，自动触发仍需 5 条门槛。

### 6.2 GREEN: 最小实现通过测试

- 仅改 handlers/client 调度逻辑。
- 不改 MemoryManager 解析与存储协议。

## 7. 验收标准

1. AI 主动触发后，本回合结束必定执行一次记忆生成。
2. 该执行不受最低轮数限制。
3. 普通自动触发仍受 5 条门槛限制。
4. 同回合多次触发只执行一次。
5. 失败不阻断主流程且不泄漏状态到下一回合。
6. 无数据库迁移与 schema 改动。

## 8. 发布与回滚

### 8.1 发布策略

1. 按 TDD 先 RED 后 GREEN 小步提交。
2. 先跑 unit，再跑 memory 相关 integration。

### 8.2 回滚策略

1. 回滚 handlers/client 的强制标记与回合末优先级逻辑即可。
2. MemoryManager 与数据层不变，无迁移回滚风险。
